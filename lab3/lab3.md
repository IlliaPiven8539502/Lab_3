## Теоретичні відомості

**Кодування** — етап розробки програмного забезпечення, на якому архітектурні рішення перетворюються на виконуваний код. Якісне кодування передбачає:

- **Модульність** — розбиття системи на незалежні компоненти з чіткими зонами відповідальності
- **Читабельність** — зрозумілі імена, коментарі, однорідний стиль
- **Принцип єдиної відповідальності (SRP)** — кожен клас відповідає лише за одну річ
- **Dependency Injection** — залежності передаються ззовні, а не створюються всередині
- **Обробка помилок** — система коректно реагує на некоректні вхідні дані

### Опис компонентів

#### `models/` — Моделі даних

Прості класи (POJO) без бізнес-логіки. Кожен клас містить лише поля, конструктори, гетери/сетери та метод `toString()` для серіалізації в JSON.

| Клас | Поля | Призначення |
|---|---|---|
| `User` | `id`, `name` | Учасник розмови |
| `Message` | `id`, `conversationId`, `senderId`, `text`, `createdAt`, `status` | Повідомлення з відстеженням статусу |
| `Conversation` | `id`, `type` | Контейнер для повідомлень |

#### `storage/` — Сховище даних

Клас `Database` відповідає виключно за читання та запис файлів. Не містить бізнес-логіки. Використовує формат **JSON Lines** — кожен рядок файлу є окремим JSON-об'єктом:

```
{"id":"abc...","name":"Аліса"}
{"id":"def...","name":"Богдан"}
```

Основні методи:

| Метод | Дія |
|---|---|
| `saveUser(user)` | Дозаписує рядок у `users.jsonl` |
| `loadUsers()` | Зчитує всі рядки, парсить у `List<User>` |
| `findUserById(id)` | Фільтрує список за id |
| `saveMessage(msg)` | Дозаписує рядок у `messages.jsonl` |
| `updateMessageStatus(id, status)` | Перечитує файл, замінює статус, перезаписує |

#### `services/` — Бізнес-логіка

Клас `MessageService` — посередник між HTTP API та сховищем. Виконує всі перевірки перед збереженням даних.

Приклад методу `sendMessage` з трьома перевірками:

```java
public Message sendMessage(String conversationId, String senderId, String text)
        throws IOException {

    // Перевірка 1: текст не може бути порожнім
    if (text == null || text.isBlank()) {
        throw new IllegalArgumentException("Текст повідомлення не може бути порожнім");
    }

    // Перевірка 2: відправник повинен існувати
    Optional<User> sender = db.findUserById(senderId);
    if (sender.isEmpty()) {
        throw new IllegalArgumentException("Користувача не знайдено");
    }

    // Перевірка 3: розмова повинна існувати
    Optional<Conversation> conv = db.findConversationById(conversationId);
    if (conv.isEmpty()) {
        throw new IllegalArgumentException("Розмову не знайдено");
    }

    // Створюємо повідомлення зі статусом "sent"
    Message msg = new Message(
        UUID.randomUUID().toString(),
        conversationId, senderId,
        text.trim(),
        Instant.now().toString()
    );
    db.saveMessage(msg);
    return msg;
}
```

#### `api/` — HTTP API

Клас `Router` приймає HTTP-запити, розпізнає маршрут за методом і шляхом, викликає відповідний метод сервісу та повертає JSON-відповідь.

| Метод | Маршрут | Обробник |
|---|---|---|
| GET | `/` | Інтерактивний веб-інтерфейс |
| POST | `/users` | Створити користувача |
| GET | `/users` | Список користувачів |
| POST | `/conversations` | Створити розмову |
| GET | `/conversations/{id}/messages` | Історія розмови |
| POST | `/messages` | Надіслати повідомлення |
| PATCH | `/messages/{id}/status` | Оновити статус (Варіант 2) |

#### `Main.java` — Точка входу

Ініціалізує компоненти у порядку залежностей (Dependency Injection):

```java
Database       db      = new Database(dataDir);  // 1. сховище
MessageService service = new MessageService(db); // 2. бізнес-логіка
Router         router  = new Router(service, port); // 3. HTTP API
router.start();
```

## Реалізований варіант — відстеження статусів

Варіант 2 реалізований через ендпоінт `PATCH /messages/{id}/status`.

Життєвий цикл повідомлення:

```
[*] ──► sent ──► delivered ──► read ──► [*]
```

- `sent` — встановлюється автоматично при створенні повідомлення
- `delivered` — клієнт надсилає підтвердження отримання
- `read` — клієнт надсилає підтвердження прочитання

## Висновок

У ході виконання лабораторної роботи №3 реалізовано повноцінний програмний код системи обміну повідомленнями на мові Java відповідно до архітектури, спроєктованої у попередніх роботах.

Застосовано такі принципи якісного кодування:
- **SRP** — кожен клас має одну зону відповідальності (моделі, сховище, логіка, API)
- **Dependency Injection** — компоненти отримують залежності через конструктор
- **Валідація на рівні сервісу** — некоректні дані відхиляються до збереження
- **Коментування** — кожен клас і метод супроводжується JavaDoc
